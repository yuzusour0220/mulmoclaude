# バグ issue の深刻度ラベル（severity）

バグ報告の issue を起票・トリアージするとき、または深刻度の判断に迷ったときに読む。

バグ報告の issue には、影響の大きさを表す**深刻度ラベルを1つ**付ける。あわせて `bug` ラベルも付ける（バグ報告だと一目で分かるように）。feat / 要望 / 検討系の issue には深刻度を付けない。

## ラベルと定義

| ラベル | 意味 | 典型例 |
| --- | --- | --- |
| `blocker` | 致命的。機能が使えない・データが失われる/壊れる、回避策なし。リリースを出せなくする問題もここ | アプリが起動できない / 保存したデータが消える |
| `major` | 重要な機能が壊れている（回避策はある） | 特定の環境・設定でコア機能が常に失敗するが、別の手順でしのげる |
| `normal` | ふつうの不具合 | 時々表示が乱れるがリロードで直る |
| `minor` | 軽微。少し不便な程度 | リンクの飛び先が誤っており、手で辿り直せば済む |
| `trivial` | ささいなもの | 誤字・わずかな見た目のズレ |

## 判断軸

深刻度は**バグの影響度だけ**で決める。対応する/しないの優先度判断は別工程 — 「早く直してほしいから上げる」「直さない見込みだから上げ下げする」といった逆算をしない。

1. **発生時の深さ** — 踏んだときに何がどこまで壊れるか（表示だけ / 機能の一部 / アプリ全体 / データ）
2. **発生条件の広さと安定性** — 全員に起きるか、特定の環境・操作の人だけか。条件が狭くても、**その条件下で常に壊れる**（当事者はその機能を安定して使えない）なら `major`。広く起きても「時々＋簡単な復旧」なら `normal`
3. **回避策の質** — 存在するか、そして**自明か**。回避策があっても、原因不明で調査に時間を溶かす類は1段上げる（原因が issue 等で文書化済みなら、このコストは支払い済みとして適用しない）
4. **データ損失・破損の有無** — 消える・壊れた状態で保存されるなら一気に blocker 寄り。表示だけで裏のデータが無事なら下げる

### 目安（較正）

- `major` は「その機能が（回避策なしには）**安定して使えない**」状態に取っておく。**時々起きる**表示・同期系の不具合で、簡単な操作（リロード等）で完全復旧しデータも無事なもの → `normal` どまり
- **セキュリティの問題**（認可の迂回・他ユーザーのデータが見える等）は、発生条件が狭くても `blocker` / `major` 級として扱う
- **実用にならないレベルの性能劣化・頻発するクラッシュ**は「機能が壊れている」と同等に扱う（機能が残っていても使えなければ同じ）
- **開発環境でしか起きない**バグ（エンドユーザーに発生経路がない）→ 症状が全損級でも原則 `normal` 以下。ただし**リリース作業そのものを止める**もの（ビルド不能・CI が恒常的に赤等）は `blocker`

## 判定例

| issue | 判定 | 決め手 |
| --- | --- | --- |
| #1915 チャット表示がターン途中で停止 | `normal` | コア機能だが壊れるのは表示だけ（セッションは裏で完了・データ無事）。発生は瞬断時のみで常時ではなく、リロードで完全復旧。「安定して使えない」までは行かない |
| #1596 パスに "test" を含むと全 API 401 | `normal` | 発生時は全 API 不通と症状は重いが、開発環境限定（エンドユーザーには起きない）で回避も明確（パス名変更）。リリースは止めない。原因調査コストは issue の文書化で支払い済み |
| #1548 インラインコードリンクが誤パスへ遷移 | `minor` | 補助機能の不具合でデータは無事・回避も自明（ファイルツリーから開けばよい）。「少し不便な程度」 |

## 更新のしかた

基準や境界に違和感が出たら、この文書を PR で更新する。境界で議論になった実例は「判定例」に追記して基準を育てる。追記するときは「決め手」欄に、症状・発生条件・回避策・データ影響が読み取れる一言を含める。
